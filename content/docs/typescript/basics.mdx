## What is TypeScript?

**TypeScript** is a _superset_ of JavaScript that adds **static typing** to the language. It helps catch errors during development instead of at runtime.

Think of it like this:

```
JavaScript + Type Checking + Modern Features = TypeScript
```

---

## Why Use TypeScript?

| Feature           | Benefit                                            |
| ----------------- | -------------------------------------------------- |
| Static Typing     | Catches bugs early                                 |
| IDE Support       | Better autocomplete, inline docs, refactoring      |
| Code Scalability  | More maintainable for large apps                   |
| Rich Ecosystem    | Integrates with React, Node, Express, etc.         |
| Future JavaScript | Supports ES6+ features with backward compatibility |

## TypeScript Setup

### 1. Install TypeScript globally:

```bash

```

### 2. Create a project:

```bash

cd ts-basics
npm init -y
tsc --init
```

### 3. Add a `.ts` file:

```bash
touch index.ts
```

### 4. Compile:

## Basic Types in TypeScript

| Type                 | Example                        |
| -------------------- | ------------------------------ |
| `string`             | `let name: string = "Diwash"`  |
| `number`             | `let age: number = 23`         |
| `boolean`            | `let isAdmin: boolean = false` |
| `any`                | `let x: any = "hello"`         |
| `unknown`            | Safer version of `any`         |
| `null` / `undefined` | Special JS values              |

### Example:

```ts
let fullName: string = "Diwash Bhattarai";
let age: number = 23;
let isDeveloper: boolean = true;
```

## Type Inference

TypeScript can **infer** the type even if you don’t write it explicitly:

```ts
let city = "Kathmandu"; // inferred as string
```

But always prefer **explicit types** for clarity and long-term maintainability.

## `any` vs `unknown` vs `never`

### `any` – Turn Off Type Checking

### Purpose:

Allows **any type of value** and disables type checking.

### Use Case:

- Used when you're migrating JavaScript or dealing with dynamic input.
- But **not recommended** in general because it bypasses the type system.

### Example:

```ts
let value: any;

value = "hello";
value = 123;
value = { foo: "bar" };

value.doSomething(); // ❌ No error at compile time, but might crash at runtime
```

---

## `unknown` – Safe Alternative to `any`

### Purpose:

Like `any`, can hold any type, but **you must narrow it** before using it.

### Use Case:

- Use it when you want to accept any type but still enforce type safety before use.
- Common in APIs, validation libraries, and `JSON.parse()`.

### Example:

```ts
let value: unknown;

value = "hello";
value = 123;

// value.toUpperCase(); ❌ Error: Object is of type 'unknown'

if (typeof value === "string") {
  console.log(value.toUpperCase()); // ✅ Safe now
}
```

---

## `never` – Represents _Nothing_

### Purpose:

Represents a value that **never occurs**.

### Use Case:

- A function that throws an error or runs infinitely.
- Exhaustiveness checking in switch or if/else chains.

### Example 1 – Function that never returns:

```ts
function throwError(message: string): never {
  throw new Error(message);
}
```

## Summary Table

| Type      | Accepts Any Value? | Can Be Used Directly? | Type-Safe? | Common Use Case            |
| --------- | ------------------ | --------------------- | ---------- | -------------------------- |
| `any`     | ✅ Yes             | ✅ Yes                | ❌ No      | Quick fixes, legacy code   |
| `unknown` | ✅ Yes             | ❌ No (must narrow)   | ✅ Yes     | API inputs, `JSON.parse()` |
| `never`   | ❌ No              | ❌ No                 | ✅ Yes     | Functions that throw       |
